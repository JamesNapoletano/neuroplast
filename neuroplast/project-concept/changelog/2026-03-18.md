# 2026-03-18
#changelog

## Summary
- Added a dedicated portability concept document that reframes Neuroplast as a portable workflow specification built around the `/neuroplast/` filesystem contract.
- Added five sequential portability phase plans covering workflow contract stabilization, metadata, capabilities and validation, environment guides, and CLI polish.
- Standardized the portability roadmap on root `ARCHITECTURE.md` as the canonical architecture artifact.
- Updated root `ARCHITECTURE.md` with the current portability roadmap direction and CLI scope boundaries.
- Added a reusable learning note on stabilizing the workflow contract before environment guides or adapter-style convenience features.
- Completed Phase 1 by adding `WORKFLOW_CONTRACT.md` to the source and installed workflow files and including it in CLI bootstrap behavior.
- Updated conceptualization, planning, execution, and README guidance so root `ARCHITECTURE.md` is the canonical architecture artifact across the workflow.
- Clarified that Obsidian wiki-links and config are optional-compatible conventions while the filesystem contract and markdown role tags remain required workflow conventions.
- Completed Phase 2 by adding `manifest.yaml` to the source and installed workflow files and including it in CLI bootstrap behavior.
- Added lightweight YAML frontmatter across source and installed instruction files using stable workflow metadata fields only.
- Documented metadata boundaries so manifest structure and core frontmatter fields are normative while review hints and tags remain advisory.
- Completed Phase 3 by adding `capabilities.yaml` to the source and installed workflow files and including it in CLI bootstrap behavior.
- Added graceful degradation rules for constrained environments and wired execution guidance to consult the advisory capability profile before acting.
- Added explicit portability validation targets in the manifest and validation rules in the workflow contract without expanding the CLI yet.
- Completed Phase 4 by adding optional environment guides for OpenCode, Claude Code, Cursor, Windsurf, VS Code + Copilot, and terminal-only workflows.
- Added a shared environment-guide template and documented `neuroplast/adapters/` as the canonical bundled location for documentation-only guidance.
- Updated manifest, workflow contract, README, and architecture docs so environment guides are clearly optional and cannot override the workflow contract.
- Updated CLI bootstrap and package architecture docs so bundled environment guides install under `/neuroplast/adapters/` in target projects.
- Completed Phase 5 by adding `neuroplast validate` for contract and metadata validation without expanding into environment orchestration.
- Updated package scripts, CLI help, README, architecture docs, and concept docs to reflect the final portability model and narrow CLI scope.
- Kept adapter functionality documentation-only while using validation to enforce contract boundaries for environment guides.
- Fixed the environment-guide index doc so validation treats it as a non-authoritative guide aligned with the canonical contract, manifest, and capability profile.
- Clarified all adapter guides with a mandatory-start sequence so agents are explicitly told to read the workflow contract, manifest, and capabilities profile before executing workflow steps.
- Added an explicit sync-impact decision gate to execution and changelog instructions so managed-file changes must record whether a new migration is required.
- Added a plan and reusable learning note documenting the rule that migration-worthy managed-file changes must be decided and, when needed, implemented in the same execution cycle.
- Reversed the package-wide migration-gate instruction changes so maintainer-only sync policy no longer affects every Neuroplast install.
- Added an optional workflow extension model with support for both bundled reusable extension scaffolding and repo-local custom extensions declared in `neuroplast/manifest.yaml`.
- Moved `package-maintainer` to a repo-local extension so migration-governance guidance remains opt-in for this repository without shipping to every install.
- Updated adapter guidance, workflow contract, manifest, CLI bootstrap, validation, README, architecture docs, and concept docs to account for active workflow extensions.
- Added a reusable learning note on keeping maintainer-specific policy out of the core workflow contract.
- Added a new managed-file backfill migration for version `1.1.1` so existing installs can receive package-managed additions introduced since the prior migration.
- Documented a seamless step-loading convention so active extensions feel automatic without changing the core workflow files.
